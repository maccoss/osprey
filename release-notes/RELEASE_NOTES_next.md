## Bug Fixes

- **Decoupled `lib_cosine` counting features from the low-norm gate** (thanks to Brendan MacLean, [PR #6](https://github.com/maccoss/osprey/pull/6)). `lib_cosine` previously returned `SpectralScore::default()` as soon as `lib_norm < 1e-10` or `obs_norm < 1e-10`, zeroing out every field in the score. The norm gate is the right call for the cosine, Pearson, and Spearman features (all three are undefined when a norm is zero and produce NaN without the guard), but the presence/counting features (`n_matched`, `consecutive_ions`, `explained_intensity`, `fragment_coverage`, `top6_matches`, `sequence_coverage`, `n_matched_b`, `n_matched_y`) come from `match_fragments` output and are independent of whether the cosine itself is computable. Restructured the gate: compute the norms, set `cosine_ok = (both >= 1e-10)`, and branch only the norm-dependent features on `cosine_ok`. Observed effect on an Astral validation run: closed 59 short/low-signal rows where mass-accuracy features clearly showed fragment matches but `consecutive_ions` and `explained_intensity` reported zero. Added regression test `lib_cosine_counting_features_survive_zero_norm`.

- **Preserved full-precision isolation-window bounds in DIA spectrum grouping** (thanks to Brendan MacLean, [PR #7](https://github.com/maccoss/osprey/pull/7)). `group_spectra_by_isolation_window` used truncated 0.1 m/z bounds as the HashMap key to tolerate floating-point noise across spectra in the same DIA window (correct), but then also emitted the truncated values as the window bounds themselves. A DIA window with real bounds `[400.05, 410.05]` was reported as `[400.0, 410.0]`, and every downstream consumer that filtered library entries by precursor m/z was using the truncated bounds. Entries sitting right at the window boundary got silently misplaced into the adjacent window. Fix: keep the truncated `(i64, i64)` as the hash key so floating-point noise can't split a window into multiple buckets, but store the full-precision bounds alongside the spectrum indices and emit the per-bucket full-precision bounds. Observed effect on a HeLa Astral file: ~2 out of ~100K entries at window boundaries (~0.002%); small in absolute count but the misplaced entries cascaded through calibration LDA fold assignment and affected several downstream q-values. Added regression test `test_group_spectra_preserves_full_precision_bounds`.

- **Fixed blib output admitting charge states that fail precursor-level FDR.** Commit `53e6a0e` (v26.2.0) replaced the blib gate from `FdrLevel::Both` (`max(precursor_q, peptide_q) <= threshold`) to `config.fdr_level` (default `Peptide`), which let all charge states of a peptide-level-passing peptide into the blib regardless of their individual precursor-level q-values. On Stellar 3-file HeLa: 50,149 precursors pass precursor-level FDR, but 58,513 were written to the blib (8,364 phantom charge states). Some precursors had no ID line in any replicate because they never passed run-level FDR in any file. Fix: two-stage blib gate. Stage 1 (peptide gate): `--fdr-level` determines which peptide identities are eligible. Stage 2 (precursor gate, always enforced): within each eligible peptide, only charge states passing `experiment_precursor_qvalue <= threshold` are included. If no charge state of a peptide passes precursor-level FDR, the best charge state is kept as a fallback representative. Also changed the `--fdr-level` default from `Peptide` to `Precursor`. Added two regression tests: `test_blib_gate_excludes_charge_states_failing_precursor_fdr` and `test_blib_gate_fallback_keeps_best_charge_state`.

- **Fixed reconciliation tolerance inflation from wrong-peak detections.** On the 44-file Seer/Floyd dataset, 36% of files had reconciliation RT tolerances > 0.4 min (max 2.26 min), effectively disabling reconciliation for those files. Root cause: `refit_calibration_with_consensus` builds calibration from `(consensus_library_rt, entry.apex_rt)` pairs where `entry.apex_rt` may be at the wrong peak from the first-pass search. Wrong apex_rt values create large LOESS residuals; the global MAD (P50 of all absolute residuals) is inflated; and the inflated MAD produces a wide tolerance that classifies wrong peaks as "Keep". Fix: (1) sigma-clipped MAD — compute initial 3-sigma clip threshold, discard outlier residuals, recompute MAD from survivors; (2) cap tolerance at the original calibration's tolerance (the reconciliation tolerance should only tighten, never loosen, through successive calibration passes). Added `abs_residuals()` accessor on `RTCalibration` and three regression tests for the sigma-clip helper.

- **Deduplicated fragment bins in `scorer.xcorr` and `xcorr_at_scan`** (thanks to Brendan MacLean, [PR #8](https://github.com/maccoss/osprey/pull/8)). Comet-style XCorr scores the dot product of an experimental preprocessed spectrum against a theoretical spectrum with unit intensity per bin. `preprocess_library_for_xcorr` already honors this by assigning `binned[bin] = 1.0` for each unique bin hit, so the fast path (`xcorr_from_preprocessed`) was correct by construction. The two slow-path entry points `scorer.xcorr()` and `xcorr_at_scan()` used a per-fragment sum that double-counted bins whenever two library fragments fell into the same bin (common on unit-resolution binning, occasional on HRAM), so direct and preprocessed XCorr disagreed for the same inputs with direct biased high on dense fragment lists. Both direct paths now iterate a visited-bin set and sum each bin's preprocessed value at most once. The accumulator was also promoted from `f32` to `f64` to match `xcorr_from_preprocessed`. Added regression test `xcorr_dedups_fragment_bin_collisions` with fragments at 500.0 and 500.5 (colliding unit-resolution bin) plus one fragment in a distinct bin.

## Performance

- **Used unit-resolution bins for calibration XCorr scorer on HRAM runs** (thanks to Brendan MacLean, [PR #3](https://github.com/maccoss/osprey/pull/3)). `run_calibration_discovery_windowed` previously selected `SpectralScorer::hram()` (~100K bins/spectrum) for HRAM mode, while `run_xcorr_calibration_scoring` (batch.rs) had long used unit-resolution bins (~2K bins/spectrum) unconditionally, noting in-code that calibration XCorr only needs to discriminate target/decoy and unit-resolution bins are dramatically cheaper. Aligned the discovery-windowed path with the batch path: both now use `SpectralScorer::new()` for calibration regardless of resolution mode. Eliminates the 100K-bin allocation on HRAM calibration runs without changing calibration output (the LDA operates on the same target/decoy rank ordering either way). Main search (`run_search`) is unchanged and still picks the resolution-appropriate scorer (HRAM for HRAM, unit-resolution otherwise).

- **Added regression tests for calibration XCorr bin choice** (thanks to Brendan MacLean, [PR #4](https://github.com/maccoss/osprey/pull/4)). Follow-up to PR #3 that extracts the calibration XCorr scorer construction into a private `calibration_xcorr_scorer` helper and adds unit tests covering both HRAM and unit-resolution paths. Both tests expect the scorer's bin count to match `SpectralScorer::new()` (~2K bins); the HRAM case additionally asserts the count is not `SpectralScorer::hram()`'s (~100K). If anyone puts `SpectralScorer::hram()` back into the calibration path, the HRAM test fails with a message pointing at `docs/02-calibration.md` and naming the helper.

## Breaking Changes

- **Changed `--fdr-level` default from `Peptide` to `Precursor`.** Users who relied on the implicit peptide-level gate will now get precursor-level filtering by default. To restore the old behavior, pass `--fdr-level peptide` explicitly. This change was made because the two-stage blib gate now enforces precursor-level FDR within each peptide regardless of the setting, making `Precursor` the natural default.
