# Osprey Release Notes (Next Release)

Working draft for the next release. Append entries here as features and fixes land on the development branch. At release time this file is renamed to `RELEASE_NOTES_v{version}.md` and the workspace `Cargo.toml` version is updated to match.

## New Features

- **Stage 5 / Stage 6 cross-implementation diagnostic dumps for OspreySharp parity.** Ten new env-var-gated TSV dumps localize where the Rust and C# (OspreySharp) ports diverge during a `--join-only` run, without affecting production behavior when the gates are unset. Each dump uses an `OSPREY_DUMP_<NAME>` / `OSPREY_<NAME>_ONLY` pair (the second exits the process after writing for fast bisection). Stage 5 covers the Percolator FDR pipeline: `OSPREY_DUMP_STANDARDIZER` (per-feature mean/std), `OSPREY_DUMP_SUBSAMPLE` (subsample membership and fold IDs), `OSPREY_DUMP_SVM_WEIGHTS` (per-fold SVM weights, bias, iteration count), and `OSPREY_DUMP_PERCOLATOR` (per-FdrEntry SVM score, PEP, and all four q-values, before compaction so decoys are present). Stage 6 covers the planning checkpoint: `OSPREY_DUMP_CALIBRATION` (loaded `library_rts` / `fitted_values` per file), `OSPREY_DUMP_PROTEIN_FDR` (first-pass picked-protein FDR per peptide), `OSPREY_DUMP_CONSENSUS` (per-peptide consensus RT and MAD), `OSPREY_DUMP_INV_PREDICT` (per-detection apex_rt / library_rt / sigmoid weight), `OSPREY_DUMP_MULTICHARGE` (per-file rescore targets keyed by stable `entry_id`, not Vec position), and `OSPREY_DUMP_REFIT` + `OSPREY_DUMP_LOESS_FIT` (refined-calibration scalar stats vs. full LOESS curve, for stats-vs-smoother bisection). All numeric columns route through a shared `format_f64_roundtrip` helper that handles NaN/inf and normalizes `-0.0 â†’ "0"` to match .NET BCL G17. Rows are sorted on stable keys (peptide, entry_id, fold_id) so cross-impl comparison scripts hash-join sort-order-agnostic.

## Bug Fixes

- **Fixed PEP non-determinism in first-pass FDR.** `compute_fdr_from_stubs` previously iterated the `targets` and `decoys` HashMaps directly when building the `winner_scores` / `winner_is_decoy` arrays for `PepEstimator::fit_default`. Rust HashMap iteration order is randomized per process, and `PepEstimator`'s internal float sums are not associative, so per-entry PEPs drifted by ~1 ULP run-to-run. Compounding this, `Kde::pdf` used `par_iter().fold().sum()` for the kernel summation â€” Rayon's work-stealing tree shape is non-deterministic, so the same KDE fit produced ~1-ULP-different bins across calls. Fixed by sorting the union of base_ids before building the winner arrays, and by switching `Kde::pdf` to a serial `iter().fold()` reduction. The performance cost is negligible (â‰Ş 1 second for ~1000 bins Ă— ~241k samples on Stellar) compared to the Percolator SVM training that precedes it. Stage 5 dumps are now byte-identical across runs of the same input.

- **Fixed `grid_search_c` tie-break to match the documented "first C wins" intent.** The grid search used `Iterator::max_by_key`, which per stdlib docs returns the **last** tied element, but the surrounding comment said "first C as tiebreaker" and the OspreySharp port already implemented first-tied. On a tie between e.g. C=1.0 and C=10.0, Rust picked 10.0 while C# picked 1.0 â€” the higher-C model has more complexity, so per-fold SVM weights drifted between implementations. Replaced with a manual scan using strict `>` so first-tied wins.

- **Fixed `count_passing_targets_svm` internal inconsistency that affected SVM hyperparameter selection.** The grid-search `count_passing_targets_svm` used a conservative `(n_decoy + 1) / n_target` q-value formula, but the sibling `select_positive_training_set` on the same Percolator iteration path uses non-conservative `n_decoy / n_target` (matching the documented `compute_qvalues` helper). The two functions evaluated the same training data against incompatible objectives, causing grid search to pick a different `best_c` from OspreySharp on borderline cases. Switched to non-conservative q-values with the standard backward monotonization pass; conservative q-values are still used for final reported FDR.

- **Guard LDA fit against single-class input.** `LinearDiscriminantAnalysis::fit` previously returned `None` silently when given an all-target or all-decoy label set: the absent-class loop iteration produced a 0-row class matrix whose covariance divides by zero, seeding `scatter_within` with NaN and causing the downstream `Gauss::solve` to return `None`. The caller couldn't distinguish this precondition violation from a genuine numerical failure. Added an explicit guard at the top of `fit` that counts targets and decoys in one pass, names the missing class in the warn-log, and returns `None` before any matrix arithmetic. Calling code (LDA per-fold calibration scoring) now logs the cause instead of a generic "training failed" error.

- **Fixed Gauss-Jordan partial pivoting and tolerance-based singularity checks.** Three correctness bugs in the Gauss-Jordan elimination used by LDA scatter-matrix solves: (1) `echelon` selected the pivot row by **signed** max, so a column like `[-4, 0]` picked the zero pivot (because `0 > -4`) and skipped the large-magnitude negative â€” partial pivoting fix uses `abs()`; (2) the zero-pivot guard used exact equality `== 0.0` instead of a `1e-12` tolerance, allowing rounding-debris pivots like `1e-18` to produce factors of `1e18` that obliterated the row; (3) `left_solved` used signed off-diagonal comparison `x > 1E-8` (a large negative passed silently), exact-equality diagonal checks (`x != 1.0 && x != 0.0` rejects `0.99999999`), and silently accepted NaN entries â€” fixed by switching all comparisons to `(x - target).abs() > TOL`, adding an explicit `is_finite` rejection, and requiring zero-rows on the left to have matching zero-rows on the right (otherwise the system is inconsistent). Mirrors fixes landing simultaneously in the OspreySharp port (ProteoWizard/pwiz#4155). Verified locally on Stellar and Astral at 1e-6 with no observable change to current production paths.

- **Fixed RT calibration JSON parser to round-trip f64 values exactly.** `serde_json`'s default fast number parser is not correctly rounded for full-precision f64 â€” the decimal `1.9140296182650374` parses to `0x3FFE9FDD85606113` under `f64::from_str` and .NET `double.Parse` (the closest representable f64) but to `0x3FFE9FDD85606114` (1 ULP higher) under serde_json's default. `ryu` writes the calibration JSON correctly, but reading back through serde_json drifted at the borderline, breaking byte-parity with OspreySharp on `--join-only` reruns. Added a custom deserializer for the three `Vec<f64>` arrays in `RTModelParams` (`library_rts`, `fitted_rts`, `abs_residuals`) that captures each element via `RawValue` and re-parses through `f64::from_str`. Calibration JSONs now round-trip exactly. Enabled the `raw_value` feature on `serde_json` at the workspace root.

- **Stage 6 calibration refit now honors `OSPREY_LOESS_CLASSICAL_ROBUST` (default on).** `refit_calibration_with_consensus` was building its `RTCalibrator` via the `with_bandwidth` / `with_outlier_retention` builders, which left `classical_robust_iterations` at the struct's `Default::default()` value of `false`. Stage 4's main calibration reads the env-var with default-on semantics, so the same pipeline run was using classical Cleveland 1979 robustness in Stage 4 and the legacy single-refresh algorithm in Stage 6 refit. OspreySharp's Stage 6 also defaults to classical-on, so Rust's refit diverged cross-impl. Switched refit to `RTCalibrator::with_config` and the same env-var read used at Stage 4. Set `OSPREY_LOESS_CLASSICAL_ROBUST=0` to disable.

- **Stage 6 reconciliation gate now fires on `--join-only` multi-file runs.** Both `reconciliation_enabled` checks were `config.reconciliation.enabled && config.input_files.len() > 1`. On `--join-only` invocations driven by `--input-scores` parquets, `config.input_files.len()` is 0, so multi-file `--join-only` runs skipped Stage 6 reconciliation entirely. Switched both gates to the post-load `per_file_entries.len() > 1`, which equals `input_files.len()` for non-`--join-only` runs (no behavior change there) and correctly reflects parquet count for `--join-only`.
